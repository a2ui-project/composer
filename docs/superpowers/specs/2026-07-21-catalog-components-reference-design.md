# Custom Catalog → Catalog Components Reference — Design

**Date:** 2026-07-21
**Branch:** `jerel/ux-update` (on Google upstream `ec76b5d`)
**Status:** Approved (design questions answered 2026-07-21)

## Goal

Give the composer a per-component **reference** for our 11-component custom
dashboard catalog — the honest, in-app equivalent of the widget-builder's
"Catalog Components" tab — with **native previews** (real `@a2ui/angular`
renders, not iframes) plus raw **source viewers** for the catalog's
definitions (`apis.ts`) and renderers (the 11 `.ts` files). Both viewers are
feasible here, unlike the gallery, because we own that source.

## Why this is distinct from the gallery

- The **gallery** (`/gallery`) documents whatever catalog the *iframe renderer*
  advertises. It can show a "Definition" (the published catalog JSON) but not
  renderer source — the renderers live in an external iframe app we don't own.
- The **custom catalog** is authored in this repo (`apis.ts` + 11 Angular
  renderers). So a reference for it can show Preview + Usage + Props **and**
  the actual definition/renderer source.

## Placement (decision: separate nav route)

A new top-level route **`/catalog-reference`** with its own sidenav entry
("Catalog Reference", icon `menu_book`), a sibling of "Custom Catalog". It
provides its own component-scoped `A2uiRendererService` + `A2UI_RENDERER_CONFIG`
(never root — instance-per-injector, same rule as the Assembled page) and
reuses `buildDashboardCatalog()`.

Route uses **no `startupGuard`** (native render, no iframe renderer needed) —
identical to the `custom-catalog` route.

## Page structure

A top view switcher (mat-button-toggle) with three faces:

### 1. Components (default)
- **Left sidebar:** the 11 components grouped by category —
  - *Layout:* Row, Column, DashboardCard
  - *Content:* Title, Badge
  - *Data Display:* Metric, PieChart, BarChart, DataTable, FlightCard
  - *Interactive:* Button
- **Detail pane** for the selected component:
  - **Preview** — the component rendered live on its own A2UI surface
    (`surfaceId = "ref-<Name>"`) via `<a2ui-v09-surface>`, using the same
    native binder as the Assembled page.
  - **Usage** — the A2UI JSON snippet (`createSurface` + `updateComponents`
    [+ `updateDataModel`]) in a read-only `<pre><code>` block, with a copy
    button (mirrors the gallery's Usage card).
  - **Props** — a `mat-table` (Name · Type · Required · Description).

The first component is auto-selected so the page never opens on an empty pane
(same principle as the gallery auto-select).

### 2. Definitions
A file list (`apis.ts`, `dashboard-catalog.ts`) + a read-only `<pre><code>`
source pane for the selected file.

### 3. Renderers
A file list (the 11 renderer `.ts` files, labeled by component) + the same
read-only source pane.

Source panes use a plain read-only `<pre><code>` block, **not** the shared
Monaco editor: the shared `<a2ui-composer-monaco-editor>` is hardwired to a
single JSON model (`a2ui://layout.json`, `language: 'json'`, one shared model
across all instances) and would neither highlight TypeScript nor host two
independent source documents. A `<pre>` is consistent with the gallery's
existing Usage block and adds no dependencies.

## Source bundling (decision: raw source viewers via prebuild codegen)

`?raw` imports are **not** viable: they work under Angular's Vite dev-server
(`ng serve`) but the `@angular-devkit/build-angular:application` (esbuild)
production build does not honor the `?raw` suffix — it would break `ng build`.

Instead, a small Node codegen script reads the source files and emits a normal
TypeScript data module (a plain `export const`), which is 100% robust in both
dev and prod:

- **Script:** `shell/scripts/generate-catalog-source.mjs`
  - Reads `shell/src/app/custom-catalog/catalog/apis.ts`,
    `.../catalog/dashboard-catalog.ts`, and the 11
    `.../catalog/components/**/<name>.ts` renderer files (excluding `.spec.ts`
    and `test-harness.ts`).
  - Writes `shell/src/app/custom-catalog/reference/catalog-source.generated.ts`
    exporting:
    ```ts
    export interface CatalogSourceFile { label: string; path: string; code: string; }
    export const CATALOG_SOURCE: {
      definitions: CatalogSourceFile[];
      renderers: CatalogSourceFile[];
    } = { /* … */ };
    ```
  - Each `code` is embedded via `JSON.stringify` of the file text (safe
    escaping). `path` is repo-relative; `label` is a friendly name.
  - Emits a banner: license header + `/* eslint-disable */` +
    `// GENERATED by scripts/generate-catalog-source.mjs — do not edit.`
- **Wiring:** `package.json` gets `"generate:catalog-source": "node scripts/generate-catalog-source.mjs"`.
  The generated file **is committed** (so the launch.json build + `ng serve`
  never depend on a prebuild hook, and the concurrent session isn't disrupted).
  Re-run the script when renderer source changes.
- **Lint/format:** add `**/*.generated.ts` to `eslint.config.js` `ignores`
  and `.prettierignore`; add `shell/scripts/**` to eslint `ignores`.

The generated module is pure data (one `export const` literal) — no functions,
no branches — so it satisfies the coverage thresholds trivially once imported
by the reference component.

## Component metadata (hand-authored)

`shell/src/app/custom-catalog/reference/catalog-reference.data.ts` — a
hand-authored table (the widget-builder's own approach with
`flight-components-data.ts`), chosen over runtime zod introspection because it
yields clean curated docs and the `usage`/`data` double as the preview payload:

```ts
export type CatalogCategory = 'Layout' | 'Content' | 'Data Display' | 'Interactive';
export interface PropDoc { name: string; type: string; required: boolean; description: string; }
export interface ComponentDoc {
  name: string;            // matches the ComponentApi name (e.g. 'FlightCard')
  category: CatalogCategory;
  description: string;
  props: PropDoc[];
  usage: unknown[];        // a complete, self-contained A2UI component tree (has a root)
  data?: Record<string, unknown>; // optional data model for the preview
}
export const COMPONENT_DOCS: ComponentDoc[] = [ /* 11 entries */ ];
```

Each `usage` tree is authored to render standalone (wrappers/children
included), so the Preview surface is valid without external context. Previews
favor literal props (no data binding needed) for simplicity; `data` is used
only where a component reads it.

Prop `type` strings are readable aliases of the zod contract: `DynamicString`,
`ChildList`, `ComponentId`, `Action`, `DynamicValue`, `number`, `string`,
enum unions like `'h1' | 'h2' | 'h3'`.

## Data flow (Preview)

Same proven pattern as the Assembled page (avoids the stale-surface bug):
- Component-scoped providers build the catalog once.
- A `builtSurfaces: Set<string>` guards each per-component surface: on first
  selection of a component, send `createSurface` + `updateComponents`
  (+ `updateDataModel` if `data`) once; thereafter the surface is just shown by
  `surfaceId`. Each component has a distinct `surfaceId` (`ref-<Name>`), so
  switching selection never re-sends `createSurface` to a built surface.

## Testing

- **catalog-reference.spec.ts** (Monaco not used, so no stub needed; but
  `<a2ui-v09-surface>` needs the component-scoped renderer providers, which the
  component supplies): view-switch toggles the three faces; component selection
  updates the detail; auto-select picks the first component; Usage renders JSON;
  source panes render `CATALOG_SOURCE` file lists and switch files.
- **composer-shell.spec.ts** — bump nav counts for the added entry.
- **Data integrity guard** — a test asserting every `COMPONENT_DOCS[i].name`
  exists in `buildDashboardCatalog()` and vice-versa (keeps docs in sync with
  the catalog).

Full suite (`corepack yarn exec vitest run`) must stay green with coverage met.

preview; category groups populated; Usage + Props correct; Definitions and
